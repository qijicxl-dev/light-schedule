
# 轻排程

## 今日讨论纪要（2026-04-16）

### 一、产品定位
我们要设计的是一款**面向入门级离散制造企业**、**依托金蝶云星空**的**轻量排程驾驶舱 + 调整台**。

它不是传统重型 APS，也不是单纯展示型看板，而是一个：
- 以**车间执行排产**为主场景
- 兼顾**老板可视化查看与简单调整**
- 围绕**已下达生产工单**进行排程
- 支持**自动初排 + 人工微调 + 急单半自动重排**
- 结果可**深度回写**到金蝶云星空的执行闭环系统

---

### 二、已确认的需求边界

#### 1. 主场景
- 以 **B：车间执行排产** 为主
- 兼顾 **D：老板可视化排产**

#### 2. 在金蝶云星空中的定位
- 选择 **C：驾驶舱 + 调整台**
- 金蝶云星空负责主数据、业务单据、基础流程
- 本产品负责看板、排程、拖拽调整、结果回写

#### 3. 第一阶段目标客户
- **离散制造企业**
- **工艺路线相对固定**
- 不做流程制造优先，不做大而全行业通用抽象

#### 4. 第一阶段核心用户
- **计划员 / PMC**
- **车间主任 / 调度员**
- 老板/厂长主要是查看结果与少量干预，不作为第一操作者

#### 5. 排程颗粒度
- 选择 **C：工单级为主，关键工序下钻**
- 平时按工单/任务进行排程
- 对瓶颈工序、关键设备、关键环节支持下钻

#### 6. 排程目标优先级
- **现场可执行优先**
- **兼顾交期**
- 不追求数学最优，而追求：
  - 计划员敢用
  - 车间愿意照着执行
  - 结果可解释、可调整

#### 7. 回写深度
- 选择 **C：深度回写执行结果**
- 本产品不仅输出建议，还要进入执行闭环

#### 8. 排程对象起点
- 选择 **C：生产工单为主，支持少量急单插入**
- 以已下达生产工单/制令单为主
- 临时急单支持插入并触发重排

---

### 三、第一阶段必须纳入的约束
第一阶段确定需要纳入以下约束：
- **A：设备 / 产线能力约束**
- **B：工序先后约束**
- **C：班次 / 日历约束**
- **D：插单 / 急单优先级约束**

在此基础上，又补充确认了一个关键约束：
- **E：上下级任务 / 上下游工单依赖约束**

也就是说，系统不能只看单张任务本身，还要保证上下游关系不被排乱。

---

### 四、排程方式
第一阶段排程方式确定为：
- **A 为主：系统自动初排 + 人工微调**
- **D 必须有：急单插入后的半自动重排**

对应含义：
- 系统先给出初始可执行排程方案
- 计划员 / 车间主任再进行拖拽、换线、顺序调整
- 急单插入后，系统要快速给出新的可执行方案，并明确影响范围

---

### 五、物料问题的处理方式
关于齐套 / 缺料问题，第一阶段确认：
- **物料齐套作为风险提醒，不作为硬约束**

即：
- 系统可以提示缺料风险、未齐套风险
- 但不直接阻止工单进入排程
- 是否继续排、是否先占位，由计划员/车间主任判断

这样做是为了在“轻量级”和“真实可用”之间取得平衡。

---

### 六、第一阶段成功标准
上线 3 个月后，最希望客户感知到的价值排序是：
1. **B：现场更稳，少乱**
2. **D：车间透明度更高**
3. **A：排程效率明显提升**

也就是说，本产品最核心的价值不是“算得多复杂”，而是：
- 插单、改单、催单来了以后，不再全靠吼
- 计划、车间、老板看的是同一套结果
- 现场调整更可控，沟通成本更低

---

### 七、关于任务级排程下的上下级顺序保证
针对“任务级排程时，工单的上下级先后顺序如何保证”的问题，今天已经明确：

#### 1. 依赖来源
第一阶段采用：
- **C：两者都要，但以 A 为主**

即：
- 主依赖来源：**金蝶云星空已有的生产工单树 / BOM层级 / 上下游工单关系**
- 补充依赖来源：**少量人工维护的前置任务关系**

#### 2. 排程规则
- 上游任务未满足前置条件时，下游任务不能正式开工
- 自动初排必须遵守依赖顺序
- 人工拖拽若破坏关键依赖，系统要阻止或强提醒

#### 3. 上下级放行规则
第一阶段采用：
- **C：默认整单完工，关键场景允许部分放行**

具体含义：
- 默认规则：上级工单整单完工后，下级才能开始
- 特殊规则：少数场景允许达到最小放行量后，下级提前开始
- 第一阶段采用配置化开关，不做复杂动态批次流转模型

---

### 八、当前推荐的产品方案方向
今天初步比较了 3 个方向：
1. 轻看板型
2. 轻引擎型
3. 执行闭环型

当前推荐方向为：
- **方案二：轻引擎型（轻量规则排程引擎 + 可视化调整台）**

原因：
- 最符合“轻量级”定位
- 最贴合“现场可执行优先、兼顾交期”的目标
- 最适合挂在金蝶云星空之上，形成清晰边界

---

### 九、已形成的核心流程草案
今天已经初步提出以下流程框架：
1. **拉取待排任务**
2. **系统自动初排**
3. **计划员/车间主任人工微调**
4. **急单插入与半自动重排**
5. **结果确认与深度回写**
6. **过程可视化与老板查看**

其中主流程暂定为：
> **自动初排 → 人工微调 → 确认回写**

急单重排则作为一条重点能力单独强调。

## 第 2 部分：核心功能模块清单

### 一、核心功能模块
我建议第一阶段围绕以下 7 个核心模块展开，避免把系统拆得过碎：

#### 1. 待排任务池
负责从金蝶云星空拉取已下达生产工单，并统一组织为待排任务集合。

第一阶段需要承载的信息包括：
- 工单号
- 产品/物料
- 数量
- 交期
- 工艺路线
- 候选产线/设备
- 上下级依赖关系
- 急单标识
- 齐套风险标识

它的作用是让计划员在正式排程前，先看清：
- 哪些任务还没排
- 哪些任务优先级更高
- 哪些任务存在风险或依赖未满足

#### 2. 轻量排程引擎
这是系统的核心能力层，负责基于规则生成“可执行的初始排程方案”。

第一阶段纳入的规则包括：
- 设备 / 产线能力约束
- 工序先后约束
- 班次 / 日历约束
- 急单优先级约束
- 上下级工单 / 任务依赖约束
- 缺料风险提醒（非硬约束）

这一模块的定位不是重型 APS 优化器，而是：
> **轻规则、强约束、可解释的排程引擎**。

#### 3. 排程调整台
这是计划员和车间主任最常用的主操作界面，用于在系统初排基础上进行人工微调。

第一阶段至少支持：
- 拖拽调整任务顺序
- 切换产线/设备
- 锁定任务不参与重排
- 调整优先级
- 查看上下级依赖链
- 对关键工序进行下钻查看

建议至少提供三类视图：
- 工单时间视图
- 资源甘特视图
- 任务依赖视图

#### 4. 约束校验与风险提醒中心
负责在自动初排、人工调整、急单重排、回写前统一进行风险与约束检查。

第一阶段重点检查：
- 资源是否冲突
- 工序顺序是否被破坏
- 是否跨班次或不可执行日历
- 是否破坏上下级依赖
- 是否导致交期风险增加
- 是否存在缺料/齐套风险

建议第一阶段用三类级别呈现：
- 红色：硬冲突，不能通过
- 黄色：风险提醒，可确认继续
- 蓝色：信息提示

#### 5. 急单插入与影响分析模块
负责处理急单/插单场景，是第一阶段的重要差异化能力。

第一阶段至少支持：
- 将急单插入指定时间窗或资源池
- 自动挪动受影响任务
- 展示受影响任务列表
- 标示延期风险和依赖影响链

重点不在“算得多复杂”，而在让用户清楚看到：
- 哪些任务被挤压
- 哪些工单可能延期
- 哪些依赖链被影响

#### 6. 回写确认与执行闭环模块
因为产品定位明确要求深度回写，所以这一模块是闭环关键。

第一阶段建议支持：
- 草稿态排程调整
- 用户确认后统一回写
- 回写状态记录
- 基础操作留痕
- 回写失败结果展示

核心原则是：
> **排程确认后回写，而不是每次拖动都实时写回。**

#### 7. 老板驾驶舱
用于满足老板/厂长的查看诉求，不以操作为主，而以全局透明度为主。

第一阶段建议展示：
- 今日/本周排程概览
- 产线/设备负荷情况
- 急单影响概览
- 延期风险工单
- 当前瓶颈资源
- 计划稳定性与异常任务数

### 二、模块优先级建议

#### P0：必须具备
- 待排任务池
- 轻量排程引擎
- 排程调整台
- 约束校验与风险提醒
- 回写确认与执行闭环

#### P1：强烈建议具备
- 急单插入与影响分析

#### P1 / P2：增强价值模块
- 老板驾驶舱

### 三、一句话总结
> **前面收任务，中间排任务，人工调任务，系统防排错，最后回写执行，并把风险和结果可视化给管理层。**

## 第 3 部分：关键数据对象与系统边界

### 一、核心数据对象

#### 1. 生产工单（Work Order）
这是系统最基础的排程主体，来自金蝶云星空，是“排什么”的起点。

第一阶段建议关注的关键字段包括：
- 工单号
- 产品/物料编码
- 产品名称
- 计划数量
- 已完工数量
- 计划开工时间
- 计划完工时间
- 交期
- 工单状态
- 所属车间
- 工艺路线标识
- 急单标记
- 优先级
- 上级/下级工单关系

#### 2. 排程任务（Scheduling Task）
这是排程引擎真正操作的对象。

第一阶段建议采用两种任务形态：
- 工单级任务：默认排程单元
- 关键工序级任务：仅针对瓶颈工序、关键设备、重点环节展开

关键字段包括：
- 任务 ID
- 关联工单号
- 任务类型
- 所属资源组
- 可分配资源范围
- 计划数量
- 最早开始时间
- 最晚完成时间
- 当前状态
- 锁定状态
- 前置任务 / 依赖任务
- 是否允许部分放行

产品定义上应明确：
> **工单是业务对象，任务是排程对象。**

#### 3. 资源（Resource）
第一阶段建议按轻量级方式分层，优先关注：
- 产线
- 设备
- 工位（必要时）
- 班组（弱约束或附属信息）
- 资源组

关键字段包括：
- 资源 ID
- 资源类型
- 资源名称
- 所属车间
- 产能参数
- 可执行工艺 / 工序范围
- 工作日历 / 班次
- 当前状态
- 瓶颈标识

#### 4. 工艺与工序（Process / Operation）
用于支撑“关键工序下钻”和“工序顺序约束”。

第一阶段的原则是：
- 普通任务：只保留工艺路线信息，不做全量精排
- 关键任务：拉出关键工序节点参与排程

关键字段包括：
- 工艺路线 ID
- 工序 ID
- 工序名称
- 顺序号
- 标准工时 / 节拍
- 资源需求
- 是否关键工序
- 前后置关系

#### 5. 依赖关系（Dependency）
用于定义工单、任务之间的前后置约束，是任务级排程可信的关键。

第一阶段来源：
- 主来源：金蝶云星空中的上下级工单 / BOM / 工单树关系
- 补充来源：人工维护的前置任务关系

关键字段包括：
- 依赖 ID
- 上游对象
- 下游对象
- 依赖类型
- 放行规则（整单完工 / 部分放行）
- 最小放行量或比例
- 是否关键依赖

#### 6. 日历与班次（Calendar / Shift）
用于定义资源在什么时间可以执行任务。

第一阶段至少支持：
- 工作日 / 非工作日
- 白班 / 夜班 / 多班次
- 停机时段
- 节假日
- 临时加班或停机调整

### 二、结果对象

#### 7. 排程结果（Schedule Result）
这是系统最终输出并用于回写的结果快照。

关键字段包括：
- 结果 ID / 版本号
- 关联任务 / 工单
- 分配资源
- 开始时间
- 结束时间
- 排程序号
- 调整来源（系统初排 / 人工调整 / 急单重排）
- 风险状态
- 回写状态
- 确认状态

这一对象必须支持可追溯和版本化，便于解释：
- 当前结果是谁确认的
- 哪次调整导致结果变化
- 哪次急单重排影响了当前版本

### 三、系统边界

#### 由金蝶云星空提供
第一阶段建议复用金蝶已有的数据与主档：
- 物料 / 产品
- 工艺路线
- 生产工单
- 车间
- 设备 / 产线基础档案
- 班次基础信息（若已有）
- BOM / 上下级工单关系
- 工单状态
- 已完工数量
- 下达状态
- 急单标识（如已有）
- 齐套 / 缺料基础状态（如可获取）

#### 由本系统维护
本系统重点维护与排程决策和执行组织相关的数据：
- 任务池状态
- 排程任务对象
- 人工锁定状态
- 排程序号
- 人工调整记录
- 急单插入记录
- 风险校验结果
- 排程版本快照
- 关键工序定义
- 资源分配规则
- 优先级规则
- 特殊依赖规则
- 部分放行配置
- 风险提醒阈值

### 四、一句话边界定义
> **金蝶云星空负责生产业务主数据与业务事实，本产品负责排程任务组织、规则计算、人工调整、风险校验、结果确认与执行回写。**

## 第 4 部分：关键交互与页面结构建议

### 一、整体页面结构建议

#### 1. 排程驾驶舱 / 主工作台
这是计划员和车间主任的主界面，建议采用三栏式或 2.5 栏式结构：
- 左侧：待排任务池
- 中间：主排程视图（工单时间轴 / 甘特图）
- 右侧：任务详情与风险面板

它需要帮助用户快速回答四个问题：
1. 还有哪些任务没排
2. 当前排程是如何分布的
3. 哪些地方有冲突或风险
4. 下一步该怎么调

#### 2. 任务详情页 / 侧边抽屉
第一阶段建议以侧边抽屉或弹层为主，而不是重页面跳转。

需要展示的信息包括：
- 工单基本信息
- 当前排程结果
- 可选设备 / 产线
- 关键工序信息
- 上下级依赖链
- 风险提示
- 调整记录

#### 3. 急单插入与重排弹层
建议单独做成独立弹层或流程页，用于承载高频但强动作的急单场景。

至少支持：
- 选择急单
- 指定插入方式（优先插队 / 指定资源 / 指定时间窗）
- 展示重排前后对比
- 显示受影响任务列表
- 标出延期风险和依赖影响链
- 确认是否执行重排

#### 4. 风险中心
建议既支持在主页面右侧显示局部风险，也支持一个全局风险视图。

重点汇总：
- 交期风险
- 资源冲突
- 依赖破坏
- 缺料提醒
- 超班次 / 跨日历排程问题

#### 5. 老板驾驶舱
建议独立成页，但更轻量，主要承载：
- 今日 / 本周排程概览
- 瓶颈资源排行
- 延期风险工单
- 急单影响概览
- 简化版甘特图或负荷图

### 二、关键交互原则
- 默认先看结果，再做动作
- 关键调整必须所见即所得
- 强约束阻止，弱风险提醒
- 尽量减少页面跳转，在主工作台内完成主要操作

### 三、第一阶段建议页面清单
- 排程主工作台
- 急单插入 / 重排弹层
- 全局风险视图
- 回写确认页 / 弹层
- 老板驾驶舱

### 四、一句话总结
> **一个主工作台承载排程与调整，几个轻辅助界面承载急单、风险、回写和老板视图。**

## 第 5 部分：回写机制、权限与风控边界

### 一、回写机制
第一阶段建议采用：
> **草稿调整 → 用户确认 → 批量回写**

不建议用户每次拖动任务都实时写回金蝶，因为那会导致数据抖动、执行依据不稳，也不利于追溯。

### 二、建议的回写流程
1. 系统形成当前排程结果
2. 用户发起确认回写
3. 系统执行回写前校验
4. 生成回写清单
5. 正式回写金蝶云星空
6. 留痕与追溯

### 三、第一阶段建议回写的内容边界
优先回写：
- 计划开工时间
- 计划完工时间
- 分配车间 / 产线 / 设备
- 排程序号 / 优先顺序
- 派工建议结果
- 任务状态（如已排程 / 待执行）
- 排程版本标识（如需要）

第一阶段不建议大范围自动改动：
- 底层工艺路线
- 主生产计划逻辑
- 销售订单承诺逻辑
- 全量库存 / 物料业务状态
- 复杂执行反馈链

### 四、权限设计
建议第一阶段按四类能力划分权限：
- 查看权限：老板 / 厂长 / 管理人员
- 调整权限：计划员 / 车间主任 / 调度员
- 发布 / 回写权限：计划负责人 / 指定 PMC / 调度主管
- 配置权限：系统管理员 / 实施顾问 / 高级计划管理人员

### 五、风控边界

#### 必须拦截的动作（硬约束）
- 破坏关键上下级依赖
- 资源硬冲突
- 排到无效日历时段
- 回写对象状态不允许更新
- 关键数据缺失

#### 可以提醒但允许确认的动作（软风险）
- 缺料 / 未齐套风险
- 可能延期风险
- 急单导致原任务顺延
- 负荷偏高但未超硬上限
- 特定任务提前 / 延后超出经验建议区间

### 六、锁定机制
第一阶段建议支持：
- 锁定任务
- 锁定资源
- 锁定时间窗

锁定后的任务在普通重排中不应被随意改动；如确需改动，应提示高权限确认。

### 七、异常与失败处理
系统必须能解释失败原因，而不是只报错：
- 回写失败要告诉用户哪些成功、哪些失败、失败原因与是否可重试
- 回写冲突时应阻止覆盖并提示刷新
- 重排失败时应说明卡在哪类约束上，并给出人工处理建议

### 八、一句话总结
> **第一阶段采用“草稿调整、确认发布、批量回写、全程留痕”的机制，并通过分级权限、硬软约束分离、锁定机制与失败可追溯，确保深度回写不会破坏现场执行稳定性。**

## 第 6 部分：第一阶段范围切分

### 一、MVP：第一阶段必须跑通的最小闭环
MVP 的目标是让入门级离散制造企业能够基于已下达工单完成基础排程、人工调整、急单处理、回写执行。

第一阶段 MVP 建议必须包含：
- 待排任务池
- 轻量排程引擎
- 排程调整台
- 急单插入与半自动重排
- 回写确认与执行闭环
- 基础风险提示


### 二、增强版：第一阶段如果资源允许，可以一起做
建议优先考虑：
- 老板驾驶舱
- 关键工序下钻视图
- 更清晰的依赖可视化
- 更完整的排程版本管理
- 更灵活的策略配置

### 三、新增能力分析层：产能负荷、粗细能力与排产建议
这一轮扩展明确，第一阶段在“已下达工单排程执行闭环”之上，增加一层**产能负荷与能力分析模块**，但仍然保持轻量，不走重型 APS 路线。

#### 1. 模块定位
新增模块的定位是：
- 接在现有排程结果之上的**分析层**
- 面向计划员 / 调度员的**判断与解释层**
- 面向老板 / 厂长的**可视化态势层**

它不替代原有排程主流程，而是增强以下关键动作：
- 自动初排前的粗能力预警
- 人工拖拽时的细能力判断
- 资源负荷与不均衡识别
- 超负荷场景下的解释型排产建议

#### 2. 四个子能力
第一阶段新增 4 个子能力：
- **负荷展示**：看清资源 / 资源组在日、周、班次、小时维度的负荷情况
- **粗能力判断**：回答“这个周期整体能不能接得住”
- **细能力判断**：回答“这个资源这个时间点能不能落位”
- **满负荷风险建议**：在高负荷或不可落位时给出解释与候选建议

#### 3. 核心口径定义
建议统一使用以下定义：

- **负荷率 = 已分配任务占用能力 ÷ 可用能力**
- **粗能力**：按日 / 周维度，比较周期总需求能力与周期总可用能力，输出“可承载 / 紧平衡 / 超负荷”
- **细能力**：按班次 / 小时维度，判断任务能否落在某资源某时段，输出“可落位 / 可落位但高负荷 / 不可落位”
- **平衡**：识别同一资源组内的过载、空闲和明显不均衡
- **排产建议**：以风险提示为主，统一输出“问题判断 / 影响范围 / 候选建议”，不做黑箱最优重排

#### 4. 页面落位
新增能力建议挂在以下位置，而不是独立做成重页面：

**排程主工作台：**
- 任务池：承担排前粗能力预警
- 甘特图 / 时间轴：承担细能力即时反馈与负荷背景显示
- 右侧详情 / 风险面板：承担解释说明与候选建议展示

**老板驾驶舱：**
- 总体负荷卡
- 瓶颈资源卡
- 负荷平衡卡
- 高风险排程卡

**轻量专题页：**
- 增加一个“资源负荷视图 / 能力分析视图”，用于从资源视角查看负荷趋势、同组差异和高负荷时段明细

#### 5. 数据对象与规则扩展
在现有工单、任务、资源、依赖、排程草稿对象之外，第一阶段建议补充以下分析层对象与字段：

**资源对象补充字段：**
- `ResourceGroupId`
- `CapacityUnit`
- `NominalCapacityPerHour`
- `EfficiencyFactor`
- `WarningLoadRate`
- `MaxLoadRate`
- `CalendarId`

**排程任务补充字段：**
- `RequiredCapacity`
- `PlannedDurationMinutes`
- `RequiredResourceGroupId`
- `PreferredResourceId`
- `EarliestStartTime`
- `LatestFinishTime`
- `Priority`
- `IsUrgent`
- `SplitPolicy`
- `CapacityBasis`

**新增分析层对象：**
- `CapacityBucket`：用于承接资源 / 资源组在 Day / Week / Shift / Hour 维度下的可用能力、已分配能力、负荷率和负荷状态
- `CapacityAssessment`：用于承接粗能力 / 细能力判断结果、主原因、影响范围及判断前后负荷变化
- `SchedulingSuggestion`：用于承接结构化建议，如改派同组资源、顺延到下一时段、人工确认加班、人工复核等

#### 6. 数据边界
第一阶段的数据边界明确如下：

**来自金蝶云星空：**
- 已下达工单
- 工序 / 工艺路线基础信息
- 数量、交期、状态
- 上下级 / 前后序关系
- 资源基础资料
- 资源组、设备、班组
- 班次 / 日历基础信息
- 确认后需要回写的排程结果

**由本系统维护：**
- 排程草稿版本
- 拖拽调整记录
- bucket 化能力快照
- 负荷率计算结果
- 粗能力 / 细能力判断结果
- 风险项
- 排产建议
- 预警阈值、均衡阈值、建议规则参数

也就是说：
- **金蝶负责业务主数据与执行事实**
- **轻量 APS 负责排程分析、模拟调整、风险解释和确认回写**

#### 7. 核心判断逻辑
第一阶段新增能力的判断逻辑建议固定为 4 类：

**粗能力判断：**
- 以资源组为主，在日 / 周周期内汇总任务需求能力和总可用能力
- 输出：`可承载 / 紧平衡 / 超负荷`

**细能力判断：**
- 以资源 + 班次 / 小时为主，先判断时间窗是否合法，再判断容量占用，再叠加依赖与风险检查
- 输出：`可落位 / 可落位但高负荷 / 不可落位`

**负荷平衡识别：**
- 在同一资源组内横向比较负荷率，识别 `平衡 / 轻度不均衡 / 明显不均衡`
- 第一阶段只识别不均衡和可能迁移空间，不自动求全局最优

**建议生成：**
- 触发场景包括粗能力紧平衡/超负荷、细能力不可落位、可落位但高负荷、明显不均衡、急单插入影响过大
- 建议优先顺序固定为：
  1. 同组资源替代
  2. 顺延到下一可用时段
  3. 人工确认加班 / 插队
  4. 人工复核

#### 8. 页面交互要求
在具体交互上，第一阶段建议补充如下要求：

**任务池：**
- 每个任务显示粗能力状态、推荐资源组、预计占用时长、急单/风险标识、建议摘要
- 让计划员在排前就知道哪些任务需要重点关注

**甘特图 / 时间轴：**
- 用背景色表达负荷热度和非工作时段
- 拖拽时即时反馈“可落位 / 可落位但高负荷 / 不可落位”
- 不可落位时要说明原因，而不是只阻止拖拽

**右侧详情 / 风险面板：**
- 除基础任务信息外，显示粗能力结论、细能力结论、当前负荷率、同组平衡状态、风险原因、影响范围和候选建议

**急单插入：**
- 先做影响分析，再展示多个候选方案
- 方案至少比较急单满足程度、对原排程冲击、资源负荷新增压力
- 不做黑箱式“一键最优重排”

**老板驾驶舱：**
- 重点看总体负荷、瓶颈资源、组内平衡和高风险排程
- 作为监控与预警视图，不承接具体拖拽调整

#### 9. 第一阶段收口原则
新增能力进入第一阶段后，必须继续保持轻量边界：

**第一阶段必须纳入：**
- 资源 / 资源组负荷展示
- 粗能力判断
- 细能力判断
- 解释型排产建议

**第一阶段只做轻量版：**
- 负荷平衡只做识别，不做自动平衡
- 资源负荷视图只做轻量专题页，不做复杂 BI
- 建议逻辑只做规则型、可解释，不做黑箱优化

### 四、后续演进：明确先不做的范围
建议明确后置的内容包括：
- 从销售订单到生产排程的全链路联动
- 全工序精细 APS
- 物料齐套硬约束
- 多工厂 / 多组织协同排程
- 高级优化算法
- 实时设备 / 报工 / IoT 驱动重排
- 自动负荷均衡
- 复杂任务自动拆分并行
- 复杂准备时间 / 换模优化
- 经营级接单能力承诺

### 五、范围判断原则
后续所有“要不要放第一阶段”的讨论，可用以下原则判断：
1. 是否直接服务“车间执行排产更稳、更透明”
2. 第一版客户不用它能否跑通闭环
3. 是否显著增加实施复杂度与主数据要求
4. 是否会把产品从“可视化分析 + 建议提示”推向“重型自动优化”

### 六、一句话总结
> **第一阶段是在“已下达工单的车间排程执行闭环”之上，增加一层轻量的产能负荷与能力分析层，让系统不仅能排、能调、能回写，还能看负荷、判能力、讲风险、给建议。**

## 第 7 部分：验收标准与“第一阶段成功长什么样”

### 一、业务价值验收
第一阶段不是为了做一个功能很多的系统，而是为了让客户明显感受到：
- 现场更稳
- 信息更透明
- 调整更可控

建议至少达到以下业务结果：
- 排程结果能真正用于车间执行
- 急单处理方式明显改善
- 计划、车间、老板看到的是同一套排程事实
- 用户对系统建立基本信任

### 二、功能闭环验收
至少应满足以下标准：
1. 已下达工单可以进入待排任务池
2. 系统可以完成自动初排
3. 用户可以进行人工微调
4. 系统可以支持急单半自动重排
5. 排程结果可以确认并回写
6. 系统可以基于排程草稿输出资源 / 资源组的日、周负荷结果，并在工作台或驾驶舱中可视化展示
7. 任务进入任务池、或急单准备插入时，系统可以输出粗能力状态：可承载 / 紧平衡 / 超负荷
8. 用户在工作台拖拽任务到某资源某时间点时，系统能够即时反馈细能力结果：可落位 / 可落位但高负荷 / 不可落位，并说明主要原因
9. 当系统判定高负荷或不可落位时，至少可以输出一条结构化建议，包含问题判断、影响范围和候选处理方式
10. 在急单半自动重排场景中，系统能够展示插入后的负荷变化、受影响任务范围，以及至少 2 类候选处理方向

### 三、使用体验验收
建议重点验证：
- 主流程是否顺畅
- 关键结果是否足够直观
- 调整反馈是否及时
- 用户是否不需要懂 APS 理论也能顺手使用
- 计划员是否能在排前就识别高风险任务，而不是排后才返工
- 用户是否能理解系统为什么判定超负荷，以及建议为什么这样给出

### 四、集成与稳定性验收
建议重点验证：
- 数据拉取是否稳定
- 回写是否可控
- 系统能否解释异常
- 排程结果是否可追溯
- 负荷、能力、建议的计算结果是否与当前草稿版本一致

### 五、第一阶段成功画像
如果试点客户能够做到以下状态，就说明第一阶段是成功的：
- 计划员每天能从金蝶拉取已下达工单，并在统一工作台完成初排和微调
- 车间主任能按同一套结果执行和调整
- 有急单时，系统能快速给出影响范围和新的执行建议
- 关键排程结果经确认后能稳定回写金蝶
- 老板可以看到当前排程、风险和瓶颈，而不是靠口头追问进度
- 计划员能直观看到哪些资源快满、哪些资源组不均衡、哪些任务需要优先处理
- 系统在高负荷场景下不只是“报红”，而是能解释原因并给出候选处理方向

### 六、一句话总结
> **第一阶段成功，不只是做出一个能排、能调、能回写的系统，而是做出一个客户愿意真正拿来排车间、处理急单、识别负荷、判断能力、统一口径、稳定回写的轻量排程产品。**
