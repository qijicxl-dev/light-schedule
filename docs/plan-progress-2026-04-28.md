## 5. ERP / 金蝶打通放在哪一步

这是当前最需要明确的一点：

> **ERP / 金蝶打通不在当前这轮主线里，而是在“前端真实联调主链路稳定之后”的下一大阶段。**

### 为什么不是现在

因为当前阶段的目标是：
1. 先把我们自己的产品闭环跑通。
2. 先确认前端页面、后端最小接口、验证链路是稳定的。
3. 先把 `/planner`、`/dashboard`、`/capacity-analysis` 的 MVP 产品表达收清楚。

也就是说，当前优先级是：

> **先把“轻排程系统自己能稳定工作”这件事做扎实，再去接“外部 ERP 真系统”。**

### ERP 打通所在阶段

建议把它视为：

## 第三阶段：ERP / 金蝶真实集成阶段

位置在：
- 最小交付包之后
- 前端真实联调主链路之后
- `/planner` 发布成功态和结果对象基础表达稳定之后

### 这一阶段会包含什么

按 `轻排程.md` 的产品定义，ERP / 金蝶真实打通至少会包含：

#### 1）回写边界定义
需要明确：
- 回写到金蝶哪些单据
- 回写哪些字段
- 由谁触发回写
- 哪个状态算“已确认但未完成写回”
- 哪个状态算“真正写回成功”

参考：
- `轻排程.md` 中“回写确认与执行闭环模块”
- `轻排程.md` 中“深度回写边界”

#### 2）结果对象扩展
当前前端已经在逐步补：
- 确认状态
- 回写状态

但 ERP 阶段还会继续扩展到：
- 结果 ID / 版本号
- 确认人
- 调整来源
- 回写留痕
- 外部系统返回状态
- 失败原因与重试策略

#### 3）后端适配层
主要会落在：
- `backend/src/main/java/com/lightschedule/web/WritebackController.java`
- `backend/src/main/java/com/lightschedule/modules/writeback/WritebackService.java`

届时很可能还需要新增：
- ERP 请求 / 响应 DTO
- 回写状态记录模型
- 外部系统适配层 / client
- 回写失败分类与映射逻辑

#### 4）前端状态表达升级
届时 `/planner` 的发布成功态大概率不再只是：
- 请求已提交
- 已确认回写
- 已进入回写队列

而会继续区分：
- 已确认，待回写
- 回写处理中
- 回写成功
- 回写失败
- 可重试 / 待人工处理
